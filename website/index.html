Day 1: 架构总览 — 三层架构 · Tool 系统 · Agent 循环

🟢 入门 · 预计时间: 1小时

Claude Code 是一个基于终端的 AI 编程助手，理解其架构是深入学习的第一步。本节从宏观视角解析三层架构、Tool 系统和 Agent 循环的核心设计。

1

三层架构：表现层 / 核心层 / 服务层

表现层负责终端 UI（Ink/React），核心层包含 Agent 循环与 Tool 调度，服务层对接 Claude API 和文件系统等外部资源。三层各司其职，通过清晰的接口解耦。

2

Agent 循环：感知 → 思考 → 行动

Agent 循环是 Claude Code 的心脏。每一轮对话都经历：接收用户输入（感知）、调用 Claude API 推理（思考）、执行 Tool 操作（行动），然后将结果反馈给模型继续循环。

3

30+ Tool 统一接口

所有工具（BashTool、FileRead、FileWrite 等）都实现统一的 ToolDef 接口，包括名称、描述、参数 schema 和执行函数。这使得新增工具只需实现一个接口。

关键洞察： Claude Code 的架构本质上是一个「Tool-Augmented LLM Agent」——模型决定调用哪个工具，工具执行后将结果返回模型，形成闭环。

💻 核心代码

// Tool 统一接口定义
type ToolDef = {
  name: string;              // 工具名称，如 "Bash"
  description: string;       // 给 Claude 模型看的描述
  inputSchema: JSONSchema;   // 参数定义
  execute: (input: unknown) => Promise<ToolResult>;
};

Day 2: 入口与启动 — CLI 解析 · 并行预加载 · 状态初始化

🟢 入门 · 预计时间: 1小时

当你在终端输入 claude 命令时，背后发生了什么？本节解析从 CLI 入口到完成初始化的完整启动流程，揭示启动性能优化的关键技巧。

1

Commander.js 参数解析

入口文件使用 Commander.js 解析命令行参数（如 --model、--print），将用户的 CLI 输入转化为结构化的配置对象。

2

并行预加载优化启动

启动时使用 Promise.all() 并行加载配置文件、检查权限、初始化 API 客户端等任务，而非串行执行，显著减少启动时间。关键模块使用动态 import() 延迟加载。

3

State 对象管理会话

初始化完成后创建全局 State 对象，包含当前会话的所有状态：消息历史、工具配置、权限设置等。State 贯穿整个应用生命周期。

关键洞察： 动态 import() 是启动优化的核心策略——只在需要时才加载模块，避免启动时加载全部 30+ 个工具的开销。

💻 核心代码

// CLI 入口简化版 (main.tsx)
import { Command } from "commander";

const program = new Command()
  .name("claude")
  .option("-p, --print", "非交互模式，直接输出")
  .option("--model <model>", "指定模型")
  .parse(process.argv);

// 并行预加载关键资源
const [config, permissions, apiClient] = await Promise.all([
  loadConfig(),
  checkPermissions(),
  initApiClient(),

Day 3: 终端 UI (Ink) — React for CLI · Box/Text · Provider

🟢 入门 · 预计时间: 1小时

Claude Code 的终端界面并非普通的文本输出——它使用 Ink 框架，将 React 的组件化开发模式带入终端。本节解析 Ink 如何将 React 组件渲染为终端文本。

1

Ink：React for CLI

Ink 是一个用 React 编写终端应用的框架。你可以用 JSX 编写组件、使用 useState/useEffect 等 Hooks，Ink 负责将虚拟 DOM 渲染为终端中的 ANSI 文本。

2

Box = div, Text = span

<Box> 是 Ink 的容器组件，类似 HTML 的 <div>，支持 Flexbox 布局。<Text> 是文本组件，类似 <span>，支持颜色和样式设置。

3

Flexbox 布局

Ink 支持 Flexbox 的核心属性：flexDirection、justifyContent、alignItems、padding 等。终端 UI 的布局方式与 Web 开发如出一辙。

关键洞察： 选择 Ink 而非传统终端库（如 blessed），让 Claude Code 团队可以复用 React 生态的组件模式和状态管理，大幅降低 UI 开发复杂度。

💻 核心代码

// 简单的 Ink 终端 UI 组件
import { Box, Text } from "ink";

function StatusBar({ model, cost }: Props) {
  return (
    <Box flexDirection="row" justifyContent="space-between">
      <Text color="cyan">Model: {model}</Text>
      <Text color="yellow">Cost: ${cost.toFixed(4)}</Text>
    </Box>
  );
}

Day 4: REPL 与输入 — 交互循环 · 斜杠命令 · Early Input

🟡 中级 · 预计时间: 1小时

REPL（Read-Eval-Print Loop）是 Claude Code 交互的核心引擎。本节深入解析用户输入如何被捕获、解析和分发，以及 Early Input 如何解决启动延迟问题。

1

REPL.tsx：核心交互引擎

REPL.tsx 是整个交互循环的入口。它管理输入框状态、处理用户提交、调度 Agent 循环，并在 Agent 返回结果后等待下一轮输入。

2

斜杠命令系统

用户输入以 / 开头时，会被路由到斜杠命令系统。内置命令如 /help、/clear、/compact 等，支持自定义扩展。命令和普通对话走不同的处理路径。

3

Early Input 预捕获

用户经常在 Claude Code 启动完成前就开始打字。Early Input 机制在启动阶段就开始监听键盘输入并缓存，初始化完成后立即使用缓存的输入，实现无缝体验。

关键洞察： Early Input 是一个优秀的 UX 设计模式——不要让用户等你准备好，而是在你准备的同时就开始接收用户输入。

💻 核心代码

// handleSubmit 简化版
async function handleSubmit(input: string) {
  // 斜杠命令走单独路径
  if (input.startsWith("/")) {
    return handleSlashCommand(input);
  }

  // 普通对话：添加用户消息，启动 Agent 循环
  addMessage({ role: "user", content: input });
  await runAgentLoop();
}

Day 5: Tool 系统基础 — ToolDef · buildTool · 生命周期

🟡 中级 · 预计时间: 1小时

Tool 系统是 Claude Code 的核心能力引擎——模型通过调用工具来读写文件、执行命令、搜索代码。本节深入解析工具的定义、构建和生命周期。

1

ToolDef：统一接口

每个工具都遵循 ToolDef 接口定义：name 标识工具，description 告诉 Claude 模型何时使用此工具，inputSchema 定义参数结构，execute 是实际执行逻辑。

2

description 是给 Claude 看的

工具的 description 不是给人看的文档，而是 prompt 的一部分——它告诉 Claude 模型什么场景该用这个工具、有什么限制、期望什么输入。写好 description 直接影响模型的工具选择准确度。

3

inputSchema 定义参数

使用 JSON Schema 格式定义工具的输入参数，包括类型、必填项、枚举值等。Claude 模型根据 schema 生成合法的工具调用参数。

关键洞察： Tool 的 description 本质上是 prompt engineering——好的描述让模型精准选择工具，差的描述导致模型误用或忽略工具。

💻 核心代码

// buildTool 工厂函数简化版
function buildTool(config: ToolConfig): ToolDef {
  return {
    name: config.name,
    description: config.description,  // 给 Claude 模型的使用说明
    inputSchema: config.inputSchema,  // JSON Schema 参数定义
    async execute(input) {
      // 权限检查 → 参数验证 → 执行 → 返回结果
      await checkPermission(config.name, input);
      const validated = validateInput(config.inputSchema, input);
      return config.handler(validated);
    },
  };
}

Day 6: 消息类型与对话流 — 5种消息 · tool_use配对 · 规范化

🟡 中级 · 预计时间: 1小时

Claude API 的对话不是简单的「你问我答」——它有 5 种消息类型，工具调用需要严格的配对协议。理解消息结构是掌握 Agent 循环的关键。

1

5 种消息类型

user（用户输入）、assistant（模型回复）、tool_use（模型请求调用工具）、tool_result（工具执行结果）、system（系统提示）。每种类型有不同的字段结构。

2

tool_use 和 tool_result 通过 ID 配对

当模型决定调用工具时，返回 tool_use 消息并附带唯一的 id。工具执行完毕后，必须返回 tool_result 消息并携带相同的 tool_use_id，API 通过 ID 将请求和结果匹配。

3

消息规范化

发送给 API 前，消息需要经过规范化处理：确保 user/assistant 交替出现、移除空消息、截断过长内容、处理图片等特殊内容类型。

关键洞察： tool_use/tool_result 的 ID 配对机制使得模型可以在一次回复中请求调用多个工具，每个工具独立配对不会混淆。

💻 核心代码

// 消息类型定义
type Message =
  | { role: "user"; content: string }
  | { role: "assistant"; content: ContentBlock[] }
  | { role: "system"; content: string };

// tool_use 和 tool_result 配对
type ToolUseBlock = {
  type: "tool_use";
  id: string;          // 唯一 ID，如 "toolu_abc123"
  name: string;        // 工具名称
  input: unknown;      // 工具参数
};

type ToolResultBlock = {
  type: "tool_result";
  tool_use_id: string; // 必须与 tool_use 的 id 匹配
  content: string;     // 工具执行结果
};

// Mini Agent CLI 核心循环
import Anthropic from "@anthropic-ai/sdk";
import * as readline from "readline";

const client = new Anthropic();
const messages: Message[] = [];

async function agentLoop(userInput: string) {
  messages.push({ role: "user", content: userInput });

  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-20250514",
      max_tokens: 4096,
      messages,
      tools: myTools,  // 你定义的工具列表

    // 没有 tool_use → 对话结束
    if (response.stop_reason === "end_turn") {
      console.log(extractText(response));
      break;

    // 有 tool_use → 执行工具 → 继续循环
    const toolResults = await executeTools(response);
    messages.push({ role: "assistant", content: response.content });
    messages.push({ role: "user", content: toolResults });
  }
}

Day 8: BashTool 深度解析 — AST 安全解析 · 权限 · 沙盒

🟡 中级 · 预计时间: 1小时

BashTool 是 Claude Code 最强大也最危险的工具——它能执行任意 shell 命令。本节解析 BashTool 如何通过 AST 解析和多层安全机制在强大与安全之间取得平衡。

1

AST 解析检测危险命令

BashTool 不是用正则表达式匹配危险命令，而是将 shell 命令解析为 AST（抽象语法树），精确识别 rm -rf、curl | sh 等危险模式，避免正则匹配的误报和漏报。

2

多层安全防御

安全不是单一防线：① AST 解析检测危险模式 → ② 只读命令白名单 → ③ 权限系统询问用户 → ④ 沙盒环境隔离。任何一层拦截都能阻止危险操作。

3

输出截断策略

命令输出可能极长（如 find /），BashTool 会截断过长输出并保留头尾，告知模型输出被截断，避免撑爆上下文窗口。

关键洞察： 安全性不能依赖单一机制——AST 解析可能遗漏新模式，权限询问可能被用户误批准，只有多层防御才能构建可靠的安全体系。

💻 核心代码

// BashTool 安全检查流程简化版
async function executeBash(command: string): Promise<ToolResult> {
  // 1. AST 解析，检测危险模式
  const ast = parseShellScript(command);
  const risks = detectDangerousPatterns(ast);

  // 2. 只读命令白名单（ls, cat, grep 等直接放行）
  if (isReadOnlyCommand(ast)) {
    return executeInSandbox(command);
  }

  // 3. 有风险 → 请求用户授权
  if (risks.length > 0) {
    const approved = await requestPermission("Bash", command, risks);
    if (!approved) return { error: "用户拒绝执行" };
  }

  // 4. 在沙盒中执行
  return executeInSandbox(command);
}

Day 9: 文件工具 — Read/Write/Edit · Diff 显示

🟡 中级 · 预计时间: 1小时

文件操作是 Claude Code 最常用的能力。三种文件工具各有定位：Read 读取、Write 整体写入、Edit 精确修改。本节解析它们的设计理念和核心实现。

1

三种文件工具各有定位

FileRead 读取文件内容并附带行号；FileWrite 创建或完整覆写文件；FileEdit 使用 old_str/new_str 精确替换文件中的指定内容，无需重写整个文件。

2

FileEdit 用 old_str/new_str 精确编辑

FileEdit 不使用行号定位（因为行号在编辑过程中会偏移），而是用文本匹配：提供要替换的原始文本（old_str）和替换后的新文本（new_str），精确且不受其他编辑影响。

3

Diff 显示与行号标记

编辑完成后，终端会以 diff 格式高亮显示变更内容（红色删除、绿色新增），让用户一目了然地看到修改了什么。FileRead 输出时附带行号，方便引用。

关键洞察： FileEdit 选择文本匹配而非行号定位是一个深思熟虑的设计——当模型在同一轮对话中多次编辑文件，前面的编辑会改变后续的行号，但文本匹配不受影响。

💻 核心代码

// FileEditTool 核心逻辑简化版
async function fileEdit(input: {
  file_path: string;
  old_str: string;
  new_str: string;
}): Promise<ToolResult> {
  const content = await readFile(input.file_path, "utf-8");

  // 确保 old_str 在文件中唯一匹配
  const matchCount = countOccurrences(content, input.old_str);
  if (matchCount === 0) throw new Error("old_str 未找到");
  if (matchCount > 1) throw new Error("old_str 匹配多处，请提供更多上下文");

  // 执行替换
  const newContent = content.replace(input.old_str, input.new_str);
  await writeFile(input.file_path, newContent);

  return { output: generateDiff(content, newContent) };
}

Day 10: Agent 循环 — QueryEngine · query() · 流式响应

🔴 高级 · 预计时间: 1小时

Agent 循环是 Claude Code 的心脏。本节深入 query.ts 中的 while(true) 循环，解析 QueryEngine 如何驱动「调用 API → 解析响应 → 执行工具 → 再次调用」的无限循环。

1

QueryEngine：每轮对话的控制器

QueryEngine 封装了一次完整对话的逻辑：管理消息列表、调用 Claude API、处理流式响应、分发 tool_use 给对应的工具执行。

2

query() 的 while(true) 循环

核心的 query() 函数是一个 while(true) 循环：每次迭代调用 API，如果响应包含 tool_use 就执行工具并将结果加入消息列表继续循环，直到模型返回 end_turn 才跳出。

3

流式响应处理

API 调用使用流式（streaming）模式，模型生成的文本和 tool_use 实时传回。这让用户能看到「Claude 正在思考」的实时输出，而非等待完整响应。

关键洞察： while(true) 不是死循环——模型的 stop_reason === "end_turn" 是自然退出条件。Agent 的智能在于模型自己决定何时停止调用工具。

💻 核心代码

// query() 核心循环简化版
async function query(messages: Message[], tools: ToolDef[]) {
  while (true) {
    // 1. 调用 Claude API（流式）
    const stream = await client.messages.stream({
      model: "claude-sonnet-4-20250514",
      messages,
      tools: tools.map(toApiTool),
    });

    const response = await stream.finalMessage();

    // 2. 检查是否需要调用工具
    const toolUseBlocks = response.content
      .filter(b => b.type === "tool_use");

    // 3. 没有 tool_use → 对话结束
    if (toolUseBlocks.length === 0) break;

    // 4. 执行所有工具，收集结果
    messages.push({ role: "assistant", content: response.content });
    const results = await Promise.all(
      toolUseBlocks.map(block => executeTool(block))
    );
    messages.push({ role: "user", content: results });
  }
}

Day 11: 上下文管理 — System Prompt · CLAUDE.md · 记忆

🟡 中级 · 预计时间: 1小时

Claude Code 的行为不仅由模型决定，更由精心构建的上下文塑造。本节解析 System Prompt 的动态组装、CLAUDE.md 的多级加载，以及上下文如何影响模型行为。

1

System Prompt 动态组装

System Prompt 不是一个静态字符串，而是动态拼接而成：基础指令 + 工具描述 + CLAUDE.md 内容 + 当前环境信息（OS、shell、cwd 等）。每次对话都根据当前状态组装。

2

CLAUDE.md 多级加载

CLAUDE.md 从多个位置加载并按优先级合并：用户级（~/.claude/CLAUDE.md）→ 父目录级 → 项目级（./CLAUDE.md）。项目级优先级最高，可以覆盖用户级的通用设置。

3

上下文收集

除了 CLAUDE.md，上下文还包括：git 状态、当前分支、最近的 commit、项目文件结构等。这些信息帮助模型理解项目背景，做出更准确的决策。

关键洞察： CLAUDE.md 是用户与模型之间的「契约」——你可以通过它告诉 Claude 项目的技术栈、编码规范、禁止事项等，让模型行为与团队要求一致。

💻 核心代码

// buildSystemPrompt 简化版
function buildSystemPrompt(state: AppState): string {

  // 基础指令
  parts.push(BASE_SYSTEM_PROMPT);

  return parts.join("\n\n");
}

Day 12: 上下文压缩 — Snip · MicroCompact · Full Compact

🔴 高级 · 预计时间: 1小时

上下文窗口是有限的资源。当对话越来越长，Claude Code 需要在不丢失关键信息的前提下压缩上下文。本节解析三级压缩策略和 token 预算管理。

1

三级压缩策略

Snip：移除工具输出中的冗余内容（如超长文件内容）；MicroCompact：对旧消息做轻度摘要，保留关键信息；Full Compact：对整个对话做深度压缩，只保留核心要点。

2

80% 阈值自动触发

当 token 使用量达到上下文窗口的 80% 时，系统自动触发压缩。这个阈值确保在压缩完成前仍有足够空间继续对话，不会因为等待压缩而中断。

3

Token 预算管理

系统实时追踪每条消息的 token 数量，包括 system prompt、用户消息、助手回复和工具结果。预算管理确保每次 API 调用都在上下文窗口限制内。

关键洞察： 压缩是一个信息取舍的过程——Snip 去除明确冗余的内容，MicroCompact 保留信息骨架，Full Compact 只保留「模型继续工作所需的最少信息」。

💻 核心代码

// 压缩决策逻辑简化版
function shouldCompress(state: ConversationState): CompressionLevel {
  const usage = state.totalTokens / state.maxContextTokens;

  if (usage < 0.5) return "none";        // 50% 以下不压缩
  if (usage < 0.7) return "snip";         // 50-70% 轻度裁剪
  if (usage < 0.8) return "micro";        // 70-80% 微压缩
  return "full";                           // 80%+ 全量压缩
}

Day 15: 权限系统 — 多层安全 · 权限模式 · 沙盒

🔴 高级 · 预计时间: 1小时

权限系统是 Claude Code 安全架构的核心。本节解析四种权限模式的设计理念、多层防御机制，以及 CLAUDE.md 中的权限规则如何与代码层面的安全检查协同工作。

1

四种权限模式

default：危险操作逐次询问用户；plan：只允许读取操作，所有写入操作需确认（最安全）；auto：自动批准大多数操作；bypass：完全跳过权限检查（仅开发环境）。

2

多层安全防御

安全检查从外到内层层递进：① CLAUDE.md 规则（禁止特定操作）→ ② 权限模式（决定是否询问）→ ③ 工具级检查（AST 解析等）→ ④ 沙盒隔离（限制系统访问）。

3

CLAUDE.md 权限规则

用户可以在 CLAUDE.md 中定义权限规则，如「永远不要删除文件」「只在 src/ 目录下编辑」等。这些规则在权限检查时被读取并强制执行。

关键洞察： plan 模式是代码审查的最佳搭档——Claude 只能读取和分析代码，所有修改操作都需要人工确认，确保每一个变更都经过人类审批。

💻 核心代码

// 权限检查流程简化版
async function checkPermission(
  toolName: string,
  input: unknown,
  mode: PermissionMode
): Promise<boolean> {
  // 1. CLAUDE.md 规则检查
  const rules = loadPermissionRules();
  if (rules.isDenied(toolName, input)) {
    return false;  // 被 CLAUDE.md 规则明确禁止
  }

  // 2. 权限模式检查
  switch (mode) {
    case "bypass": return true;           // 跳过所有检查
    case "auto":   return true;           // 自动批准
    case "plan":
      return isReadOnlyTool(toolName);    // 只允许读取
    case "default":
      if (isReadOnlyTool(toolName)) return true;
      return await promptUser(            // 询问用户
        `允许 ${toolName} 执行此操作？`
      );
  }
}

Day 19: 成本与性能 — 成本追踪 · 启动优化 · Token 预算

🟡 中级 · 预计时间: 1小时

AI Agent 的运行成本不可忽视——每次 API 调用都消耗 token 费用。本节解析 Claude Code 如何追踪成本、优化启动性能，以及利用 Prompt Cache 降低开销。

1

实时成本追踪

每次 API 调用后，系统记录输入/输出 token 数量并按模型定价计算费用。累计费用实时显示在终端状态栏中，让用户对当前会话的花费一目了然。

2

并行预加载启动优化

启动时使用 Promise.all() 并行加载配置、检查权限、初始化 API 客户端。配合动态 import()，首次交互延迟从数秒降至毫秒级。

3

Prompt Cache

Claude API 支持 Prompt Cache：System Prompt 和消息前缀在多次调用之间可以被缓存，避免重复处理相同的 token，降低延迟和费用（缓存 token 费用约为原价的 10%）。

关键洞察： Prompt Cache 是 Claude Code 成本优化的最大杠杆——System Prompt 和对话历史在每轮都重复发送，缓存后只需支付首次的 token 费用。

💻 核心代码

// 成本追踪实现简化版
interface CostTracker {
  inputTokens: number;
  outputTokens: number;
  cacheReadTokens: number;
  totalCost: number;
}

function updateCost(tracker: CostTracker, usage: ApiUsage): CostTracker {
  const inputCost = usage.input_tokens * PRICE_PER_INPUT_TOKEN;
  const outputCost = usage.output_tokens * PRICE_PER_OUTPUT_TOKEN;
  // 缓存 token 费用约为原价的 10%
  const cacheCost = usage.cache_read_tokens * PRICE_PER_INPUT_TOKEN * 0.1;

  return {
    inputTokens: tracker.inputTokens + usage.input_tokens,
    outputTokens: tracker.outputTokens + usage.output_tokens,
    cacheReadTokens: tracker.cacheReadTokens + usage.cache_read_tokens,
    totalCost: tracker.totalCost + inputCost + outputCost + cacheCost,
  };
}

Day 20: 多 Agent 系统 — Team · Swarm · Coordinator · Worktree

🔴 高级 · 预计时间: 1小时

单个 Agent 的能力有上限——真正复杂的项目需要多个 Agent 协作。本节解析多 Agent 系统的两种模式（Team 和 Swarm），以及 Git Worktree 如何隔离并行操作。

1

Team 模式：Coordinator 分层协作

Team 模式有一个 Coordinator（协调者）负责任务分解和分配。Coordinator 将大任务拆分为子任务，分配给多个 Worker Agent，收集结果后整合。适合有明确层级的复杂项目。

2

Swarm 模式：对等 Agent 网络

Swarm 模式中所有 Agent 是对等的，没有中心协调者。Agent 之间通过共享状态（如文件系统）进行隐式协作。适合松耦合的并行任务。

3

Git Worktree 隔离并行操作

多个 Agent 同时修改代码会产生冲突。Git Worktree 为每个 Agent 创建独立的工作目录（共享同一个 .git 仓库），每个 Agent 在自己的分支上工作，最后合并。

关键洞察： Git Worktree 是多 Agent 并行编码的关键基础设施——它解决了「多个 Agent 同时编辑同一个文件」的根本冲突问题。

💻 核心代码

// Coordinator 模式简化版
async function coordinatorLoop(task: string, workerCount: number) {
  // 1. Coordinator 分解任务
  const subtasks = await decompose(task);

  // 2. 为每个 Worker 创建独立的 Git Worktree
  const worktrees = await Promise.all(
    subtasks.map((_, i) =>
      exec(`git worktree add ../worker-${i} -b worker-branch-${i}`)
    )
  );

  // 3. 并行启动 Worker Agent
  const results = await Promise.all(
    subtasks.map((subtask, i) => {
      const worker = new AgentWorker({
        cwd: worktrees[i].path,  // 各自独立的工作目录
        task: subtask,
      });
      return worker.run();
    })
  );

  // 4. 合并所有 Worker 的结果
  for (const worktree of worktrees) {
    await exec(`git merge ${worktree.branch}`);
    await exec(`git worktree remove ${worktree.path}`);
  }
}

// 完整 Agent CLI 入口 + 循环 + REPL
import Anthropic from "@anthropic-ai/sdk";
import * as readline from "readline";

const client = new Anthropic();

// 工具定义
const tools = [
  { name: "ReadFile", /* ... */ },
  { name: "WriteFile", /* ... */ },
  { name: "Bash", /* ... */ },
];

// Agent 循环
async function agentLoop(messages: Message[]) {
  const costTracker = { totalCost: 0 };

  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-20250514",
      max_tokens: 8192,
      system: buildSystemPrompt(),  // Day 11: 上下文管理
      messages,
      tools,
    });

    // Day 19: 成本追踪
    updateCost(costTracker, response.usage);

    if (response.stop_reason === "end_turn") {
      console.log(extractText(response));
      break;
    }

    // Day 5/8/9: 工具执行
    messages.push({ role: "assistant", content: response.content });
    const results = [];
    for (const block of response.content) {
      if (block.type === "tool_use") {
        // Day 15: 权限检查
        await checkPermission(block.name, block.input);
        const result = await executeTool(block);
        results.push({ type: "tool_result", tool_use_id: block.id, content: result });
      }
    }
    messages.push({ role: "user", content: results });
  }

  console.log(`\n总成本: $${costTracker.totalCost.toFixed(4)}`);
}

// REPL 交互循环
const rl = readline.createInterface({ input: process.stdin, output: process.stdout });
const messages: Message[] = [];

function prompt() {
  rl.question("You> ", async (input) => {
    if (input === "/quit") { rl.close(); return; }
    messages.push({ role: "user", content: input });
    await agentLoop(messages);
    prompt();  // 继续等待下一轮输入
  });
}

prompt();
